# Database Design

---

# 1. Introduction

The Smart Home IoT Platform uses MySQL as its primary relational database for storing application data.

The database is designed using normalization principles to ensure data consistency, scalability, maintainability, and extensibility while supporting future platform enhancements.

Spring Data JPA and Hibernate are used as the Object Relational Mapping (ORM) framework for interacting with the database.

---

# 2. Database Overview

The database stores all persistent information required by the Smart Home IoT Platform.

## Current Entities

- Device
- Activity
- Health Metric
- Sensor Reading

## Planned Entities

- Project Investment
- User
- Role
- Automation Rule
- Notification
- Camera Snapshot
- Audit Log
- Refresh Token

---

# 3. Database Design Principles

The database follows the following design principles.

## Normalization

Data redundancy is minimized through normalized entity design.

---

## Data Integrity

Primary keys, foreign keys, unique constraints, and validation rules ensure consistent and reliable data.

---

## Scalability

The schema supports future modules without requiring significant structural changes.

---

## Maintainability

Each table represents a single business entity with a clearly defined responsibility.

---

## Auditability

All entities inherit common audit fields from the BaseEntity class.

Audit fields include:

- createdAt
- updatedAt

# 5. Database Schema

## 5.1 Device

### Purpose

Stores information about every registered smart home device.

### Table

`devices`

### Fields

| Field | Description |
|---------|-------------|
| id | Primary Key |
| name | Device name |
| deviceCode | Unique device identifier |
| type | Device type |
| status | Current device status |
| motionStatus | Motion detection status |
| ipAddress | Device IP address |
| wifiStrength | Wi-Fi signal strength |
| firmwareVersion | Installed firmware version |
| lastSeen | Last heartbeat timestamp |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp |

### Relationships

- One Device → Many Sensor Readings
- Future: One Device → Many Activities

---

## 5.2 Activity

### Purpose

Stores activities generated by IoT devices.

### Table

`activities`

### Fields

| Field | Description |
|---------|-------------|
| id | Primary Key |
| deviceName | Device name |
| type | Activity type |
| title | Activity title |
| message | Activity description |
| activityTime | Event timestamp |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp |

### Relationships

Currently independent.

Future versions will reference the Device table using a foreign key.

---

## 5.3 Health Metric

### Purpose

Stores backend application health information.

### Table

`health_metrics`

### Fields

| Field | Description |
|---------|-------------|
| id | Primary Key |
| label | Metric name |
| value | Metric value |
| status | Health status |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp |

### Relationships

Currently independent.

---

## 5.4 Sensor Reading

### Purpose

Stores environmental sensor readings received from IoT devices.

### Table

`sensor_readings`

### Fields

| Field | Description |
|---------|-------------|
| id | Primary Key |
| device | Associated device |
| temperature | Temperature |
| humidity | Humidity |
| battery | Battery percentage |
| signalStrength | Wi-Fi signal strength |
| readingTime | Reading timestamp |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp |

### Relationships

Many Sensor Readings belong to one Device.

---

## 5.5 Project Investment *(Sprint 5)*

### Purpose

Stores all project-related investments including hardware, infrastructure, networking, software, and development tools.

### Table

`project_investments`

### Fields

| Field | Description |
|---------|-------------|
| id | Primary Key |
| itemName | Purchased item |
| category | Investment category |
| brand | Manufacturer or brand |
| quantity | Number of items |
| unitPrice | Price per item |
| totalPrice | Total purchase amount |
| purchaseDate | Purchase date |
| vendor | Vendor or supplier |
| orderReference | Order reference number |
| status | Purchase status |
| notes | Additional notes |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp |

### Relationships

Currently independent.

Future versions may optionally reference Devices.

# 7. Constraints

The database enforces integrity using the following constraints.

## Primary Keys

Every table contains an auto-generated primary key.

---

## Unique Constraints

Current unique fields include:

- Device.deviceCode

Future unique fields may include:

- User.email
- Role.name

---

## Foreign Keys

Current foreign keys include:

- SensorReading.device_id → Device.id

Future foreign keys include:

- Activity.device_id
- Automation.device_id
- Notification.user_id

---

## NOT NULL Constraints

Mandatory fields are protected using NOT NULL constraints.

---
