# Database Design

---

# 1. Introduction

The Smart Home IoT Platform uses MySQL as its primary relational database for storing application data.

The database is designed using normalization principles to ensure data consistency, scalability, and maintainability while supporting future platform enhancements.

Spring Data JPA and Hibernate are used as the Object Relational Mapping (ORM) framework for interacting with the database.

---

# 2. Database Overview

The database stores all persistent information required by the Smart Home IoT Platform.

Current entities include:

- Device
- Activity
- Health Metric
- Sensor Reading

Future entities may include:

- User
- Automation Rule
- Notification
- Camera Snapshot
- Audit Log

---

# 3. Database Design Principles

The database follows the following design principles.

## Normalization

Data redundancy is minimized through normalized entity design.

---

## Data Integrity

Primary keys, foreign keys, and constraints ensure consistent and reliable data.

---

## Scalability

The schema supports future modules without requiring significant structural changes.

---

## Maintainability

Each table represents a single business entity with a clearly defined responsibility.

---

## Auditability

All entities inherit common audit fields from the BaseEntity class.

These include:

- createdAt
- updatedAt

# 5. Database Schema

## 5.1 Device

### Purpose

Stores information about every registered smart home device.

### Table

`devices`

### Fields

| Field           | Description                |
| --------------- | -------------------------- |
| id              | Primary Key                |
| name            | Device name                |
| deviceCode      | Unique device identifier   |
| type            | Device type                |
| status          | Current device status      |
| motionStatus    | Motion detection status    |
| ipAddress       | Device IP address          |
| wifiStrength    | Wi-Fi signal strength      |
| firmwareVersion | Installed firmware version |
| lastSeen        | Last heartbeat timestamp   |
| createdAt       | Record creation timestamp  |
| updatedAt       | Record update timestamp    |

### Relationships

One Device can have multiple Sensor Readings.

---

## 5.2 Activity

### Purpose

Stores activities generated by devices.

### Table

`activities`

### Fields

| Field        | Description               |
| ------------ | ------------------------- |
| id           | Primary Key               |
| deviceName   | Device name               |
| type         | Activity type             |
| title        | Activity title            |
| message      | Activity description      |
| activityTime | Event timestamp           |
| createdAt    | Record creation timestamp |
| updatedAt    | Record update timestamp   |

### Relationships

Currently independent.

Future versions may reference Device using a foreign key.

---

## 5.3 Health Metric

### Purpose

Stores backend application health information.

### Table

`health_metrics`

### Fields

| Field     | Description               |
| --------- | ------------------------- |
| id        | Primary Key               |
| label     | Health metric name        |
| value     | Metric value              |
| status    | Health status             |
| createdAt | Record creation timestamp |
| updatedAt | Record update timestamp   |

### Relationships

Currently independent.

---

## 5.4 Sensor Reading

### Purpose

Stores environmental sensor readings received from IoT devices.

### Table

`sensor_readings`

### Fields

| Field          | Description               |
| -------------- | ------------------------- |
| id             | Primary Key               |
| device         | Associated device         |
| temperature    | Temperature value         |
| humidity       | Humidity value            |
| battery        | Battery percentage        |
| signalStrength | Wi-Fi signal strength     |
| readingTime    | Sensor reading timestamp  |
| createdAt      | Record creation timestamp |
| updatedAt      | Record update timestamp   |

### Relationships

Many Sensor Readings belong to one Device.

---

# 6. Entity Relationships

The current database relationships are:

## Device → Sensor Reading

Relationship Type:

One-to-Many

```
Device (1)
      │
      │
      ▼
Sensor Reading (N)
```

Future relationships will include:

- Device → Activity
- User → Automation
- Automation → Device
- Device → Camera Snapshot
- User → Notification

---

# 7. Constraints

The database enforces data integrity using the following constraints.

## Primary Keys

Every table contains an auto-generated primary key.

---

## Unique Constraints

Current unique fields include:

- Device.deviceCode

---

## Foreign Keys

Current foreign keys include:

- SensorReading.device_id → Device.id

---

## NOT NULL Constraints

Mandatory fields are protected using NOT NULL constraints.

---

# 8. Indexing Strategy

Current indexing includes:

- Primary Keys
- Foreign Keys
- Unique Device Code

Future indexes may be introduced for:

- Device Status
